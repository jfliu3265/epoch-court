# 《万象净界：纪元王庭》产品需求文档

> 工作名：万象净界：纪元王庭（Epoch Court）  
> 文档类型：产品需求文档 / 游戏设计总纲（PRD + High-level GDD）  
> 版本：0.1  
> 日期：2026-07-19  
> 状态：概念基线，待原型验证  
> 平台：Web，桌面浏览器优先，移动浏览器渐进适配

---

## 1. 执行摘要

《万象净界：纪元王庭》是一款原生 Web 的单人幻想冒险游戏。玩家进入一个被“噪蚀”冻结的多世界帝国，通过探索、战斗、解谜、经营、竞速、节奏、塔防等一组组短时长小游戏，逐步净化领土、重建王庭，并让世界的画面、声音、交互和玩法规则发生可感知的“时代演化”。

它不是一个小游戏列表，而是一个由许多玩法共同驱动的统一游戏世界：

- 所有小游戏共享同一名玩家角色、存档、王庭、资源体系和主线故事。
- 每个小游戏属于一个“界域”，会永久改变界域生态、居民状态或王庭能力。
- 玩家解锁的“时代透镜”可以带回旧地图，让旧内容产生新路线、新解法和新奖励。
- 游戏从极简、残缺的表现开始，随着净化进度逐步获得色彩、音乐、粒子、光影、立体空间与更复杂的游戏规则。
- Web 特效服务于玩法提示、世界变化和情绪反馈，不作为单纯的视觉堆砌。

长期愿景是形成一个可持续扩充的“游戏帝国”：9 个以上界域、30 个以上小游戏、跨界域事件、赛季内容和可选的异步社区能力。开发必须从一个可验证的垂直切片开始，通过统一的小游戏模块协议逐步扩张。

## 2. 名称校正与借鉴边界

### 2.1 关于 “Everland / 净化之地”

根据本次检索，你描述的作品很可能是 Shiro Games 的《Evoland》，中文常译《进化之地》，而不是 “Everland / 净化之地”。该作最鲜明的体验是：玩家推进游戏时，图像风格、音效、战斗方式和 RPG 机制也随之演化。

如果你指的是另一款确实叫 Everland 的作品，后续可补充链接或截图，本方案仍可调整。

### 2.2 可以借鉴的抽象体验

- 进度不仅解锁数值，也解锁表现力和游戏规则。
- 玩家持续期待“下一次世界会如何变化”。
- 通过亲自游玩感受游戏媒介与玩法的演变。
- 同一段冒险可以容纳不同类型的战斗、探索和小游戏。

### 2.3 必须保持原创的部分

- 不复刻《Evoland》的故事、角色、地图、美术、关卡或具体演化顺序。
- 不以“游戏史模仿展览”为主，而以“净化多世界、建设共同王庭”为主。
- 演化不是单向从像素到 3D；已解锁时代可以切换、叠加并参与解谜。
- 小游戏不是一次性桥段，而是王庭经济、角色成长和界域修复的长期组成部分。

## 3. 产品定位

### 3.1 一句话描述

一款会随着玩家净化世界而不断获得新画面、新声音和新规则的 Web 冒险游戏；数十个小游戏共同建成一座活着的幻想王庭。

### 3.2 类型组合

- 主类型：动作冒险 RPG、世界修复、轻度经营。
- 内容结构：主线探索 + 小游戏群 + 王庭建设 + 收藏。
- 单局长度：小游戏 2–8 分钟；主线章节 15–30 分钟。
- 总体节奏：随开随玩，短局有收益，长时间游玩也有连续叙事。
- 首发模式：单人、离线优先、无强制登录。
- 后续能力：云存档、异步挑战、好友王庭参观；实时 MMO 不在早期范围内。

### 3.3 目标玩家

核心玩家：

- 喜欢《Evoland》式机制演化惊喜的玩家。
- 喜欢塞尔达式探索、轻动作、解谜和收集的玩家。
- 喜欢多种玩法但不想在多个独立游戏之间切换的玩家。
- 喜欢世界从荒芜变繁荣、基地逐渐成长的玩家。
- 希望直接打开网页即可游玩的轻中度玩家。

次级玩家：

- 对 WebGL、像素艺术、低多边形、Shader 特效感兴趣的视觉体验玩家。
- 喜欢短局挑战、排行榜和成就的玩家。
- 喜欢发现隐藏文本、世界观和角色支线的剧情玩家。

### 3.4 产品承诺

玩家每完成一次重要净化，至少看到以下一项永久变化：

1. 世界表现发生改变；
2. 获得一种新规则或操作能力；
3. 王庭出现新建筑、居民或功能；
4. 旧区域产生新的可探索内容；
5. 一个小游戏与另一个小游戏发生系统联动。

## 4. 设计支柱

### 4.1 世界随玩家而演化

演化必须是可见、可听、可操作的，而不只是等级数字上涨。色彩、分辨率、镜头、音乐层、物理规则、敌人行为和交互方式都可以作为奖励。

### 4.2 多玩法，一个世界

每个小游戏都必须回答三个问题：

- 它属于哪个界域，为什么在这里存在？
- 它给王庭或主线带来什么永久影响？
- 玩家从其他玩法带来的能力，如何在这里产生作用？

无法回答这三个问题的小游戏不进入正式版本。

### 4.3 净化不是抹除，而是恢复多样性

反派力量“噪蚀”会把不同世界压成单调、重复、没有选择的状态。玩家的净化让颜色、物种、声音、历史和规则重新共存。主题上避免“消灭异类”的含义，强调修复、记忆与多样性。

### 4.4 炫酷必须可读、可控、可降级

- 战斗特效首先传达方向、范围、伤害类型和时机。
- 高强度闪烁、镜头摇晃、色差和模糊均可降低或关闭。
- 低配置模式保留玩法信息，削减粒子、后处理和动态分辨率。
- WebGPU 是增强路径，不是进入游戏的前提。

### 4.5 小步扩张成大帝国

先做出一个完整而精致的 20–30 分钟垂直切片，再增加界域。所有新内容复用统一协议、输入层、存档层、奖励层和质量预设，避免每个小游戏成为孤立项目。

## 5. 世界观与叙事设定

### 5.1 世界设定

世界由一座漂浮在“界海”中央的王庭和无数法则不同的界域组成。过去，王庭通过“纪元之心”允许各种文明互相交换知识、音乐、生态与游戏。一次被称为“大静默”的事件令纪元之心破裂，名为“噪蚀”的力量开始把所有界域冻结成重复、失色的残片。

玩家是从王庭废墟中醒来的“拾界者”。玩家能够听见破碎规则的回声，并通过完成当地仪式、挑战和小游戏收集“源律”，重新编织世界。

### 5.2 主要冲突

表面目标是击退噪蚀、修复纪元之心；深层冲突是：无限扩张、统一秩序是否也会制造另一种静默。后期玩家会发现噪蚀并非单纯怪物，而是旧王庭为避免世界冲突而创造的“绝对稳定协议”。

### 5.3 玩家身份

- 玩家不是天生的国王，而是通过修复与选择成为王庭的守护者。
- 玩家可以为角色命名、选择外观轮廓与代词；能力不绑定性别。
- 玩家早期没有复杂职业，随着界域恢复才获得战斗、创造、节奏和策略系能力。

### 5.4 核心角色原型

- **弥光**：王庭引导精灵，负责教程；自身也从单色光点逐渐获得形体与记忆。
- **塔洛**：修复师，管理王庭工坊，代表技术与效率。
- **苔婆婆**：生态记录者，反对只追求城市扩张，代表生命与多样性。
- **零号执政官**：旧稳定协议的人格化界面，主要对手；其动机具有合理性。
- **界域同行者**：每个界域一名核心伙伴，可入住王庭并改变小游戏能力。

### 5.5 叙事结构

- 第一幕：唤醒——恢复颜色、声音和王庭门厅。
- 第二幕：织界——发现多个界域并建立小游戏之间的资源循环。
- 第三幕：争律——各界域对王庭未来制度产生冲突。
- 第四幕：复庭——玩家决定王庭是中央帝国、松散联盟，还是流动档案馆。
- 终局后：继续净化隐藏区域、挑战高难版本、建设个性化王庭。

## 6. 游戏总体结构

### 6.1 三层空间

1. **纪元王庭**：长期成长的中心 Hub，展示玩家全部成就。
2. **界域地图**：带有探索、战斗、NPC、任务和环境谜题的章节世界。
3. **小游戏节点**：嵌入界域叙事的短局玩法，可重复挑战并产生长期收益。

### 6.2 核心循环

```text
进入界域
  → 探索并发现噪蚀节点
  → 完成一个或多个小游戏/战斗/谜题
  → 获得源律、居民信任和界域材料
  → 净化地块并恢复视觉/声音/规则
  → 返回王庭建设建筑、研发时代能力
  → 使用新能力重访旧界域，发现新路线
  → 解锁下一个界域与主线抉择
```

### 6.3 单次游戏循环

适合 5–15 分钟游玩：

- 从王庭领取一个明确目标。
- 选择一个界域入口。
- 完成一次短局挑战或一个探索节点。
- 立即获得可见奖励。
- 在安全点自动保存，可直接关闭页面。

### 6.4 长期循环

- 完成界域章节，恢复王庭区域。
- 收集时代能力并构建个人“源律编组”。
- 提升小游戏评价，解锁变体规则。
- 收集居民、服装、陈设、音乐和世界档案。
- 通过跨界域组合打开隐藏结局与终局挑战。

## 7. “庞大游戏帝国”的内容架构

### 7.1 内容规模分层

| 阶段 | 王庭 | 界域 | 小游戏 | 目的 |
|---|---:|---:|---:|---|
| 垂直切片 | 1 个门厅 | 2 个小型界域 | 4 个 | 证明演化、净化与跨玩法循环 |
| Alpha | 1 个可建设城区 | 3 个界域 | 8 个 | 证明内容管线和留存循环 |
| 1.0 | 1 座完整王庭 | 6 个主界域 | 18 个以上 | 完成主线和多结局 |
| 长期帝国 | 多城区和活动岛 | 9 个以上界域 | 30 个以上 | 持续内容、挑战与社区 |

### 7.2 界域规划

| 界域 | 视觉演化 | 核心玩法 | 小游戏示例 | 带回王庭的能力 |
|---|---|---|---|---|
| 原初荒原 | 1-bit → 8/16-bit → 高清像素 | 俯视探索、轻战斗 | 史莱姆闪避场、遗迹推箱、采集冲刺 | 色彩、音效、基础攻击、地图 |
| 森语庭院 | 剪纸 → 水彩 → 动态植被 | 生态解谜、种植、寻物 | 花粉连线、萤火节奏、藤蔓迷宫 | 生长、攀附、生态建筑 |
| 蒸汽峡湾 | 等距像素 → 低多边形机械城 | 塔防、运输、装配 | 列车调度、齿轮塔防、机械臂拼装 | 自动化、磁力、生产链 |
| 幻潮群岛 | 平面波纹 → 流体扭曲 → 立体海面 | 航行、钓鱼、节奏 | 风帆竞速、深海声呐、潮汐鼓点 | 水流、浮力、动态音乐层 |
| 星轨竞技场 | 向量线框 → 霓虹 2.5D → 高速 3D | 竞速、弹幕、Boss Rush | 光轨竞速、重力弹幕、无人机追逐 | 冲刺、时间减速、轨迹传送 |
| 梦纸书库 | 文字界面 → 纸片剧场 → 梦境空间 | 卡牌、叙事、逻辑谜题 | 句子重排、纸牌战术、墨迹潜行 | 改写标记、记忆回溯、对话洞察 |
| 方构遗城 | 体素草图 → 低模 → 可破坏构造 | 平台、建造、物理 | 方块攀登、桥梁建造、坍塌逃生 | 构造、拆解、地形编辑 |
| 深空蜂巢 | 点阵雷达 → 粒子星云 → 空间折叠 | 编队、资源策略、生存 | 舰队编队、信号译码、星云采集 | 分身、远程指挥、空间门 |
| 镜像王都 | 汇合全部风格 | Roguelite、混合 Boss、终局选择 | 时代裂隙、法则轮盘、王庭保卫战 | 自定义规则与终局形态 |

上述表格是长期内容版图，不代表首个版本一次性制作全部内容。

### 7.3 小游戏类别配额

为避免帝国内容偏科，1.0 目标至少包含：

- 4 个动作/战斗类；
- 3 个解谜/逻辑类；
- 2 个节奏/音乐类；
- 2 个竞速/反应类；
- 2 个经营/生产类；
- 2 个策略/塔防类；
- 2 个探索/平台类；
- 1 个跨类型终局玩法。

### 7.4 小游戏统一接入协议

每个小游戏以独立模块接入，并声明：

- 唯一 ID、版本、所属界域、内容包大小；
- 使用的渲染器：DOM / Canvas 2D / Phaser / Three.js；
- 所需输入动作，而不是直接绑定具体按键；
- 首次进入教程、单局时长、暂停与恢复策略；
- 难度参数、胜负条件、评分计算；
- 允许带入的时代能力和伙伴能力；
- 奖励上限、失败补偿与反刷规则；
- 存档数据结构及迁移版本；
- 资源预载、资源释放、音频总线和渲染质量档位；
- 无障碍能力：减弱动态、色盲符号、字幕、单手模式等。

小游戏退出时必须释放事件监听、计时器、粒子、纹理、音频节点和 GPU 资源。任何时刻最多保留一个重型游戏渲染器处于活动状态。

## 8. 核心系统需求

### 8.1 净化系统

每个界域由多个污染地块构成。完成地块目标后触发“净化演出”，至少包含：

- 地表材质和颜色由噪点状态过渡到当地风格；
- 环境声或音乐声部加入；
- 新 NPC、资源或道路出现；
- 地图完成度提升；
- 一项玩法规则被恢复或强化。

净化演出可跳过；重复游玩不强制播放完整动画。

### 8.2 时代透镜

玩家可在特定安全区域或谜题中切换已解锁的表现/规则层：

- **像素层**：显示隐藏网格、旧通道和可破坏砖块。
- **水彩层**：让液体、植物和情绪痕迹显形。
- **机械层**：展示机关电路、磁力点和生产路径。
- **霓虹层**：显示速度轨道、时间窗口和能量连线。
- **梦纸层**：让文本、记忆和对话选项成为可操作物体。
- **方构层**：把部分地形转换为可搬运、可旋转构件。

透镜既改变视觉，也改变碰撞、可交互对象或时间规则。不能只做滤镜换色。

### 8.3 王庭建设

王庭承担六种职责：

1. 展示被拯救的居民、生态和纪念物；
2. 提供能力研发、装备编组和外观收藏；
3. 把各小游戏产物转化为长期建设成果；
4. 提供明确的下一目标和界域入口；
5. 承载角色支线、世界抉择和结局变化；
6. 成为未来活动、社区参观和 UGC 的扩展接口。

首批建筑：纪元门厅、修复工坊、森息温室、演奏台、竞速码头、档案馆。

### 8.4 进度与能力

玩家成长分为四条轴：

- **拾界者等级**：基础生命、容错和通用插槽；避免纯数值碾压。
- **时代谱系**：解锁透镜、操作和跨玩法能力。
- **界域信任**：解锁当地故事、居民与小游戏变体。
- **王庭繁荣度**：解锁建筑、装饰、快捷功能和新界门。

时代能力采用有限插槽编组，鼓励搭配，而不是把所有能力永久堆在同一套操作上。

### 8.5 经济系统

初版最多三种主要资源：

- **光尘**：通用软资源，用于常规建设和升级。
- **源律核心**：章节级进度资源，只通过首次净化和关键任务获得。
- **界域材料**：每个界域一类主题材料，用于当地建筑、外观与伙伴支线。

设计约束：

- 垂直切片不引入付费、抽卡、体力或广告强制观看。
- 不使用十余种碎片货币制造虚假复杂度。
- 重复小游戏的收益设置软上限，但首次完成和个人最高分始终有价值。
- 若未来商业化，优先一次性购买、扩展包或纯外观；不得出售战力优势。

### 8.6 任务系统

- 主线任务：引导界域顺序和世界抉择。
- 界域任务：恢复当地生态、设施和居民关系。
- 王庭委托：在不同小游戏之间形成组合目标。
- 精通挑战：追求时间、无伤、连击或特殊规则。
- 每日/每周内容只在上线运营阶段考虑，不作为单机体验的进度门槛。

### 8.7 伙伴系统

伙伴提供探索能力和小游戏规则修改，但不替代玩家操作。例：

- 弥光可标出一次隐藏互动；
- 机械伙伴可在塔防开局提供一座临时装置；
- 潮歌伙伴可把节奏失误转化为较轻惩罚；
- 梦纸伙伴可在对话中保存一次选择分支。

### 8.8 收藏与图鉴

- 界域生态图鉴；
- 音乐层和可播放曲目；
- 时代外观、角色服装、王庭陈设；
- Boss 与噪蚀档案；
- 被玩家选择改变过的“世界年表”。

## 9. 首次用户体验（FTUE）

### 9.1 前 15 分钟体验目标

1. 页面打开后快速显示单色世界和“唤醒世界”按钮。
2. 用户点击按钮，同时满足浏览器音频启动所需的用户手势。
3. 玩家只拥有移动能力，在无声 1-bit 场景中寻找第一个光点。
4. 第一次净化恢复环境音和一种主色。
5. 第二次净化恢复攻击/互动和简单粒子。
6. 完成第一个 2–3 分钟闪避小游戏。
7. 世界从狭小画面扩展到全屏高清像素，并显示王庭远景。
8. 自动保存，玩家明确知道下一个目标。

### 9.2 教学原则

- 一次只教一个动作。
- 使用环境和角色动作示范，减少长文本弹窗。
- 第一次失败立即重试，不扣资源。
- 键鼠、触屏和手柄提示根据最近输入自动切换。
- 玩家可以在设置中重看教程和查看动作图。

## 10. Web 炫酷特效调研与采用建议

### 10.1 结论

最适合本项目的不是“全程重 3D”，而是多层次渲染：DOM 负责清晰界面，Phaser/Canvas/WebGL 负责 2D 与 2.5D 玩法，Three.js 负责少量高价值 3D 场景和演出，Shader 与音频系统贯穿所有时代。WebGL 2 为基础覆盖路径，WebGPU 仅用于高档增强。

### 10.2 特效矩阵

| 特效 | 适合场景 | 玩法价值 | 首发优先级 | 性能与降级策略 |
|---|---|---|---|---|
| GPU 粒子、火花、落叶、星尘 | 净化、技能、天气、奖励 | 显示方向、范围和世界复苏 | P0 | 数量分档；低档使用对象池和贴图序列 |
| 拖尾、丝带、光轨 | 冲刺、竞速、剑击、弹幕 | 传达速度、轨迹和命中窗口 | P0 | 降低采样点和透明层数 |
| Bloom 泛光 | 魔法核心、霓虹界域、传送门 | 突出高能量物体 | P1 | 仅选择性泛光；低档关闭或用光晕贴图 |
| 像素化、抖色、CRT、扫描线 | 原初时代、回忆、噪蚀 | 直接参与时代切换 | P0 | 单次全屏 Pass；避免过强色差和闪烁 |
| Dissolve 溶解/重构 | 净化地块、敌人消散、场景切换 | 将“修复规则”视觉化 | P0 | 噪声纹理复用；低档减少边缘光 |
| Displacement 位移扭曲 | 水面、热浪、传送门、梦境 | 提示可穿越/不稳定区域 | P1 | 限定局部区域，避免全屏持续使用 |
| 动态灯光、法线贴图、阴影 | 洞穴、机械城、Boss 战 | 提供探索线索和攻击预警 | P1 | 限制动态光数量；阴影分辨率分档 |
| 多层视差、景深层 | 2D/2.5D 探索 | 低成本制造世界纵深 | P0 | 预合成远景；低档减少层数 |
| Outline 轮廓与边缘高亮 | 可交互物、目标、低模角色 | 提高可读性与无障碍 | P0 | 可用材质/二次绘制，低档保留 |
| 流体/波纹反馈 | 幻潮群岛、鼠标触碰、声呐 | 让水和声音成为可操作规则 | P1 | 使用简化高度场或局部 RenderTarget |
| 体积雾、光柱、尘埃 | 王庭大厅、森林、遗迹 | 塑造氛围和空间层次 | P2 | 用深度雾、Billboard 和烘焙光替代真体积 |
| 屏幕震动、时间停顿、冲击波 | 重击、Boss 阶段变化 | 增强打击感 | P0 | 强度可调；支持 reduced motion |
| 音频可视化/节拍驱动 Shader | 节奏界域、音乐台、Boss | 视觉与节奏规则同步 | P1 | 分析器采样降频；非节奏模式可关闭 |
| 程序噪声、生成地形/纹理 | 噪蚀、云雾、星云、地图变体 | 减少素材重复，支持动态世界 | P1 | 预烘焙缓存；移动端降低 octave |
| CSS/SVG 微动效 | 菜单、地图线、徽记、卡片 | 保持 UI 清晰且富有生命 | P0 | 遵循 reduced motion；不用于重型战斗场景 |

### 10.3 暂不推荐作为核心的特效

- 全场景实时光线追踪或路径追踪；
- 持续开启的屏幕空间反射、重景深和高质量运动模糊；
- 大量透明粒子叠加造成的 Overdraw；
- 依赖 WebGPU 才能运行的核心玩法；
- 全屏强色差、快速闪烁和不可关闭的镜头晃动；
- 同时保留多个 WebGL/WebGPU Canvas 和大型 3D 场景常驻内存；
- 为“炫”而使用不传达任何玩法信息的过长转场。

### 10.4 特效作为成长奖励

表现层本身纳入奖励系统：

- 开局：单色、低分辨率、无混响、极少动画；
- 初期：颜色、环境音、相机缓动、基础粒子；
- 中期：动态光、Shader 材质、多层音乐、2.5D；
- 后期：选择性 3D、计算粒子、空间音频、时代叠层；
- 终局：玩家可组合时代风格，但始终保留清晰的战斗信息。

## 11. 美术与音频方向

### 11.1 统一视觉语言

虽然各界域风格不同，以下元素保持统一：

- 净化能量使用青绿、金色和暖白的连续色谱；
- 噪蚀使用低饱和紫黑、重复网格和破碎采样；
- 所有可交互对象具有一致的轮廓脉冲语言；
- UI 使用“王庭档案 + 发光源律线”的组合，不随小游戏完全换皮；
- 角色比例、轮廓识别和动画节拍建立统一规范。

### 11.2 音频演化

- 每个界域音乐拆成节奏、低音、和声、旋律、环境五个可动态混合层。
- 净化地块会永久加入一个声部或移除噪声层。
- Web Audio 用于滤波、混响、距离衰减、空间声像和节拍同步。
- 页面失焦自动降低或暂停音频；恢复时平滑淡入。
- 音乐、环境、音效、语音分别控制音量。
- 所有重要语音与歌词提供字幕；重要音效提供可选视觉提示。

## 12. UI、输入与可访问性

### 12.1 页面结构

- 启动/继续页面；
- 游戏主画布；
- 最小化 HUD：生命、当前能力、目标、互动提示；
- 王庭地图与界域地图；
- 任务、图鉴、时代谱系、设置；
- 暂停与快速退出；
- 加载界面必须显示真实进度或清晰的加载阶段，不使用假进度。

### 12.2 输入抽象

所有玩法只使用统一动作：

- 移动；
- 主行动；
- 次行动；
- 闪避/冲刺；
- 切换时代透镜；
- 伙伴能力；
- 暂停/返回。

具体小游戏可组合这些动作，但不得随意增加十余个临时按键。

### 12.3 设备支持

- P0：键盘 + 鼠标；
- P0：触屏虚拟摇杆/直接触控；
- P1：标准 Gamepad API 手柄；
- P1：键位重映射和手柄死区调整；
- P2：震动反馈，在浏览器和设备支持时启用。

### 12.4 可访问性要求

- “减少动态”选项：关闭强镜头晃动、长拖尾、视差和高频转场；
- 闪光强度与频率安全限制；
- 三套色盲辅助方案，关键信息同时使用形状/图标；
- 字幕大小、背景透明度和文本速度可调；
- 可暂停所有非联网玩法；
- 允许跳过已看过的剧情和净化演出；
- 提供瞄准辅助、节奏宽容、伤害减免等独立辅助项；
- 不以关闭辅助为成就前提；高难挑战可单独标注规则。

## 14. 非功能需求

### 14.1 性能预算

以中档笔记本和近三年中端手机为验证设备：

- 桌面 High 目标 60 FPS；移动 Medium 目标 45–60 FPS；Low 不低于稳定 30 FPS。
- 首屏尽快显示可交互启动界面；初始应用壳压缩后目标不超过 2 MB。
- 首个可玩内容包压缩后目标不超过 12 MB；其余界域按需加载。
- 加载时不得一次解码全部纹理、音乐和 3D 模型。
- 单个常规场景建议 GPU 资源预算：桌面 512 MB 内，移动 256 MB 内。
- 移动端常规 Draw Call 目标低于 150，桌面低于 300；以实际 Profiling 为准。
- 粒子必须对象池化或 GPU 化；离屏与不可见对象停止更新。
- 支持分辨率缩放，避免盲目按最高 devicePixelRatio 渲染。

### 14.2 兼容性

- 支持当前稳定版 Chrome、Edge、Firefox、Safari。
- WebGL 2 为主渲染基线；不支持时提供清晰说明或有限 Canvas 2D 序章。
- WebGPU 只作为增强，必须能回退至 WebGL 2。
- Service Worker、云存档等能力在 HTTPS 下启用。
- 移动端首发推荐横屏；竖屏显示旋转提示及安全的设置/存档入口。

### 14.3 稳定性与数据安全

- 每个净化节点、小游戏结算和王庭建设后自动保存。
- 存档写入采用临时记录 + 校验 + 原子切换，保留最近三个恢复点。
- 存档结构包含 schemaVersion，升级必须提供迁移和回滚测试。
- WebGL context lost 后尝试恢复并回到最近安全点。
- 内容包加载失败不得损坏全局存档。
- 正式版目标：无崩溃会话率大于 99.5%，已知路径零存档丢失。

### 14.4 隐私

- 单机核心不要求账号。
- 分析数据默认最小化，清晰告知并允许关闭。
- 不采集聊天、通讯录、精确位置等与游戏无关的数据。
- 未成年人相关运营和商业化必须在后续单独合规评审。

## 15. 建议技术架构

### 15.1 技术栈方向

- TypeScript + Vite：应用与构建基础。
- React 或同类 DOM UI 层：菜单、档案、设置、王庭管理界面。
- Phaser：2D/2.5D 场景、Tilemap、输入、碰撞、粒子和大多数小游戏。
- Three.js：王庭远景、低多边形界域、终局与少量高价值 3D 演出。
- Web Audio API：统一音频图、动态音乐、效果与空间化。
- IndexedDB：本地存档、内容缓存元数据和可恢复快照。
- Service Worker：PWA、离线壳、章节资源缓存与版本切换。
- Web Worker / OffscreenCanvas：在兼容路径上分离重计算或部分渲染。

具体库版本在技术原型阶段锁定；首轮不同时引入 Phaser 与 PixiJS 两套重叠 2D 引擎。

### 15.2 分层原则

```text
Web Shell / UI
  ├─ 路由、设置、账号、可访问性
  ├─ Game Host：场景生命周期、加载、暂停、质量档
  ├─ Shared Core：输入、存档、经济、任务、事件、随机种子
  ├─ Render 2D：Phaser 场景与特效
  ├─ Render 3D：Three.js 场景与后处理
  ├─ Mini-game Modules：按需加载的玩法包
  └─ Content：界域、对话、任务、奖励的声明式数据
```

游戏规则不得直接依赖 React 组件；小游戏不得直接写全局存档，而应通过标准结算事件请求核心层变更。

### 15.3 能力检测与渐进增强

启动时生成设备能力档案：

- WebGPU/WebGL 版本与扩展；
- 最大纹理尺寸、浮点纹理、压缩纹理能力；
- CPU 核心提示、设备内存提示、屏幕像素比；
- OffscreenCanvas、Worker、Gamepad、触控；
- `prefers-reduced-motion` 与系统对比度偏好。

质量档不仅影响分辨率，还影响阴影、粒子、后处理、音频卷积、植被和背景更新频率。

### 15.4 资源策略

- 应用壳、序章、各界域分别打包。
- 纹理使用 Atlas 和适合平台的压缩格式，并提供兼容回退。
- 音频分短音效与流式音乐；避免把整张原声一次载入内存。
- 3D 使用 glTF/GLB、网格压缩和纹理压缩。
- 进入界域前预取，离开后按引用计数释放。
- 所有资源具有哈希、版本、体积和所属模块元数据。

### 15.5 未来后端

垂直切片不要求后端。进入云存档与异步挑战阶段后，再引入：

- 账号/匿名身份；
- 存档版本与冲突合并；
- 成绩和幽灵数据校验；
- 内容清单与灰度发布；
- 基础反作弊；
- 运营配置。

实时联机、聊天、公会和 MMO 世界需要独立立项，不能顺手加入当前架构。

## 16. 垂直切片范围

### 16.1 必须包含

- 单色序章和“唤醒世界”交互；

- 20–30 分钟完整可玩。
- 性能、兼容、存档、恢复、加载和可访问性测试通过。
- 新玩家无需口头指导即可完成第一次净化并理解王庭目标。

---

## 18. 成功指标

垂直切片阶段以可用性和技术质量为主，不以商业收入为目标：

- 80% 测试玩家能在 3 分钟内完成第一次有效互动；
- 70% 以上能在无人工提示下完成第一次净化；
- 70% 以上完成一个小游戏后愿意主动进入第二个玩法；
- 80% 以上能说出“小游戏会帮助王庭/世界恢复”，而不是认为它们是独立列表；
- 首次 15 分钟内至少出现三次被玩家主动注意到的世界演化；
- 中端目标设备保持设定帧率，连续切换场景无明显内存增长；
- 刷新、关闭标签页和版本升级测试中不丢失已确认进度；
- 无障碍设置能真实改变游戏表现，不是占位开关。

进入线上运营后再定义 D1/D7 留存、章节完成率、小游戏复玩率、付费等指标。

## 19. 主要风险与应对

| 风险 | 影响 | 应对 |
|---|---|---|
| “庞大”导致范围失控 | 长期无法产出可玩版本 | 以垂直切片为唯一首目标；每次只扩一个界域 |
| 小游戏像独立合集 | 缺乏情感和长期目标 | 强制接入故事、王庭变化、跨玩法能力和共同经济 |
| 多引擎造成包体和内存膨胀 | 首屏慢、移动端崩溃 | 单一活动重型渲染器、按需加载、严格释放、先验证再引入 |
| 特效遮挡战斗信息 | 炫但不可玩 | 特效强度规范、危险轮廓、色盲符号、战斗可读性评审 |
| WebGPU 覆盖不足 | 部分浏览器无法进入 | WebGL 2 基线，WebGPU 仅增强，所有能力可降级 |
| 移动端触控不适配 | 大量潜在用户流失 | 统一动作少而清晰；每个小游戏必须经过触控验收 |
| 多种美术风格割裂 | 产品缺少品牌感 | 统一净化色谱、轮廓语言、角色比例、UI 与音乐主题 |
| 存档随版本损坏 | 严重信任损失 | schema 迁移、恢复点、导入导出、自动化迁移测试 |
| 内容产能不足 | 帝国无法持续扩张 | 声明式内容、小游戏模板、资源预算和模块验收清单 |
| 玩法数量多但深度浅 | 新鲜感后迅速流失 | 每个小游戏至少有精通层、规则变体和跨能力组合 |

## 20. 待确认的产品决策

这些问题不阻塞垂直切片构思，但应在 M0–M1 期间决定：

1. 正式中文名、英文名和品牌视觉。
2. 主角是完全自定义，还是具有固定形象与对白。
3. 目标美术更偏精致像素、手绘幻想，还是低多边形 3D。
4. 首发是否完整支持手机，还是桌面首发后补移动。
5. 1.0 采用一次性买断、免费序章 + 买断完整版，还是完全免费。
6. 王庭建设偏装饰展示，还是包含轻度生产链。
7. 主线调性偏温暖童话、神秘寓言，还是较深沉的文明主题。

当前默认决策：精致像素 + 2.5D 为主、桌面优先兼容移动、单人买断导向、温暖但不幼稚的幻想寓言。

## 21. 调研依据与参考链接

以下资料于 2026-07-19 查阅，用于确认产品灵感和 Web 技术边界：

- [Evoland 官方网站](https://evoland.shirogames.com/)
- [Evoland 官方 Steam 页面](https://store.steampowered.com/app/233470/Evoland/)
- [MDN：WebGPU API](https://developer.mozilla.org/en-US/docs/Web/API/WebGPU_API)——WebGPU 可提供现代 GPU 渲染和计算能力，但仍不是所有主流浏览器的统一基线。
- [MDN：WebGL 最佳实践](https://developer.mozilla.org/en-US/docs/Web/API/WebGL_API/WebGL_best_practices)——扩展与设备上限需要检测；应控制像素预算、Draw Call、同步阻塞和显存使用。
- [MDN：OffscreenCanvas](https://developer.mozilla.org/en-US/docs/Web/API/OffscreenCanvas)——可将部分 Canvas 渲染或计算移出主线程。
- [MDN：Web Audio API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Audio_API)——支持模块化音频图、动态效果、精确时序和空间声像。
- [MDN：Service Worker API](https://developer.mozilla.org/en-US/docs/Web/API/Service_Worker_API)——支持离线体验、资源缓存和后台更新策略。
- [MDN：Gamepad API](https://developer.mozilla.org/en-US/docs/Web/API/Gamepad_API)——浏览器可统一读取手柄连接、按键与轴输入。
- [PixiJS：Filters / Blend Modes](https://pixijs.com/8.x/guides/components/filters)——展示模糊、颜色、噪声、位移、自定义 Shader 和混合模式等 2D 后处理能力。
- [Phaser：ParticleEmitter](https://docs.phaser.io/api-documentation/class/gameobjects-particles-particleemitter)——粒子发射器适合构建可配置、可池化的 2D 游戏特效。
- [Three.js：UnrealBloomPass](https://threejs.org/docs/#UnrealBloomPass)——可用于选择性的高亮泛光，但应作为分档后处理能力使用。
